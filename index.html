How to watch video from your computer on every device at home without downloading it.[in two steps]

Why ?

I usually watch movies and read books from my android phone and tablet. So I decided, it would be great if I could use my main hard disk and share it to the network. I came across this method accidentally. And it showed itself very useful.

So, You will Need

  1. Linux computer. Preferably Debian
  2. Android device
  3. Standart wifi router
  4. A little patience

Let's go on. Step one. Installation.

You will definitely need nginx server. It showed itself pretty decent on even old computers. Install it like this. sudo apt-get install nginx You will probably need ffmpeg file converter. Install it like this sudo apt-get install ffmpeg You will need media player. Install vlc media player on your android device.

Step two. Configuring nginx

Let's have our hands a little bit dirty. Execute next commands sudo -s cd /etc/nginx/sites-enabled touch home-shared Put in file home-shared configuration like this: server { listen 80; location / { autoindex on; root /home/mik/shared; } } Where root is absolute path to your shared folder It would go like this echo "server { listen 80; location / { autoindex on; root /home/mik/shared; } }" > home-shared Restart your nginx server by: /etc/init.d/nginx restart In your browser, go to 127.0.0.1. You must see something like this.

Now, You must try to see this with another computer in your network. Your ip address you can see like this sudo ifconfig | grep "inet addr" Select that one wich starts with 192.168.1. Go to your android device. Or to another computer. And type into address bar that ip. Like this.

If you see directory content. Congratulations. If you don't, try disabling your firewall. Now You can put any video into that directory. And vlc from your android device will play it on the fly.

In the next part of this tutorial I will tell how to convert fast any video, so you can feed it to vlc without any problems.

I always can be wrong, or not having known something important. If you find an error, please tell me at mklimoff222 at gmail.com.

Depta for data scraping

Hello My dear reader. Today I am to tell about great library that should definitely been known more

Have you ever wondered how you can extract data from web automatically without RegExp or xpath based parsers like lxml.

State the problem

Imagine, You've got to scrape data from time series, forum posts or shop catalogues. Take for example this one. You have to extract every data item and do it continuosly e.g this day, tomorrow, next week e.t.c. Here are the problems that you will definitely come across with xpathes or regexps:

  • Change in site markdown. (Another branch, move of table e.t.c)
  • Change in data format. Was $, changed to USD. Dot was removed, or million other things.

DEPTA to the rescue

DEPTA, which is stated Data Extraction Based on Partial Tree Alignment algorithm was published in Web Data Extraction Based on Partial Tree Alignment paper by Yanhong Zhai and Bing Liu. I definitely recomment read it if you are interested in data scraping. It is based on the assumption, that meaningful repeating parts of web page will share common HTML structure between them.

How does it work

In a very simplified way the algorithm works like that:

  1. Find parent HTML node whose children subtrees are similar between themselves using Edit distance between trees
  2. Align each subchild of children subtree in column, that each element in that column share common metrics. Like xpath to this element in subtree.

Why typescript

Recently, I changed my main work environment in browser from coffeescript to typescript. Here I am to tell why. And about my work environment.

Coffee advatages over typescript

  • Lots of syntax sugar (Code in coffeescript is generally 30% shorter than typescript)
  • Indentation based folding
  • ... That's it

Typescript advantages over Coffee

  • Clear output. Principle is simple. Just strip typedeclarations and you Do get clean javascript
  • They intend to support es6(Coffeescript just does not bother about it)
  • Tooling
  • Module system (it is also a great disadvantage, see below)
  • Typesystem. nuff said

Library support

DefinitelyTyped is a very good repository for many js libraries. And if You do not have particular library there, you can just write it yourself. As I have written it for completely just in 5 minutes. // Type definitions for complete.ly // Project: http://complete-ly.appspot.com/ // Definitions by: VbifRkbvjd <https://github.com/VbifRkbvjd> // Definitions: https://github.com/borisyankov/DefinitelyTyped declare module completelyInterfaces { interface ICompletely { (el:HTMLElement,style?:any):IcompletelyInstance } interface IcompletelyInstance { options:string[] startFrom:number setText(data:string):void getText():string onChange:(data:string)=>void repaint():void hideDropDown():void wrapper:HTMLElement prompt:HTMLElement input:HTMLInputElement hint:HTMLInputElement dropDown:HTMLElement } } declare var completely: completelyInterfaces.ICompletel You see that this is pretty straightforward. And, sometimes, might work as library documentation too.

My work environment regarding typescript.

It is strange to not to use IDE capabilities in a static typed language. IDE like support in emacs is quite good via emacs-tss. But it has two disadvantages.

  1. flymake is deprecated (and checking is done via it).
  2. As a consequence of 1. any large enough file hangs

I tried to remove flymake from emacs-tss, and replace it with flycheck. But it seems that emacs-tss is heavily depend on flymake, and doing so will require major refactoring of this mode.

So I decided to switch from emacs to Intellij Idea. It has quite good support for typescript. But it also has two disadvantages

  1. Program correctness checking is weird.
    • It, sometimes, show error, when compiler compiles
    • It, sometimes, does not show error, although compiler points that error.
  2. IDEA hangs on React. Whenever you type React in your ts file, IDEA hangs, until you remove React from your file.

What I dislike about typescript

The one thing, that is definitely wrong in typescript is module system.

There are two concepts:

  • Internal modules
  • External modules

The difference between them is that external modules are about to work with require.js or browserify, when internal modules are about to glue your js files into one. You cannot use internal and external modules at the same time. And there are more intricate delicacies, like, you cannot glue module from different files if you use external system. And You can if you use internal. So, for this I was going to kill someone, untill I made a hack solution

Hack solution(No young and sensitive coders allowed)

Here is the schema I came, when I bump my head enough. Assume that I am working in node-webkit. So here it is:

  • Dependencies: deps.js
  • Compiler-ouput(one file compiled with internal module system): SomeSystemGlued.js

And deps.js contain something like: window.Q = require("q") window.R = React // because of "love" idea has to React(hangs) window.NWGUI = require("nw.gui") window._ = require("lodash") window.path = require("path") window.fs = require("fs-extra") // e.t.c And then, every subsystem just use these from related d.ts files. As they are just globals.

My evolution from dynamic to static typing.

I have been "dynamic" for the most of my career in programming(Which is not a lot,for about 3 years of actual development in python and js.)

Being dynamic was not because of some conscious decision.

Just, when you start programming(having no CS background) dynamic typing seems less mundane, and more appealing for someone outside of development.

Now, nearly a month ago I switched my preferences to static typing languages, and here I am to explain why.

During my work, I found out,and, I sure, most of software engineers do, that programming mostly is not about crunching numbers, but about connection in general.

I would formulate this software development principle as knowledge about connection and composition.

Let me elaborate.

When You are writing a new program it is about 80% of connection other people code in a new way than about writing some genuinely new code. So that is why, program language matter less than having good libraries for any possible case.

This principle directly applies to static typing. Because static typing is about connecting things. And compiler will tell you, if you connected them wrong. For example:

  • It is not about withdraw(account_id:number,sum:number).
  • It is about withdraw(account:Account,sum:MoneyManager).

In the second case, you defined what Account is, and what MoneyManager is.

And withdraw will only work on accounts, and MoneyManagers.

So, when You forgot, what is withdraw about, You can always look at signature, look at definition of account and sum. It allows you to write deeper programs, you are not restricted in connecting things just by numbers and strings. And you will never mistake pipes. And if you will, compiler will tell you about it. I deliberately do not mention obvious:

  • IDE support
  • ease of understanding
  • speed of compiled program

Because everybody already knows that.

Here goes architectural overview

In settings I've got navigation struct, like this: { "hello.html": {"date":"2015.02.28","tags":{"Hello","Contacts"},"posted":True}, "firstBlogPost.html": {"date":"2015.02.28","tags":{"hour-of-coding", "python","jinja2"},"posted":True}, "testPost.html": {"date":"2015.02.28","tags":{"hour-of-coding", "python","jinja2"},"posted":True}, "howToUseMarkDown.html":{"date":"2015.02.28","tags":{"markdown-tutorial","notForPeople"}, "posted":True}, } Here key is a category, and set tags contains possible categories for this kind of blogpost.
